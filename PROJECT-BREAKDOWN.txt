BenHowardCV — Complete Project Breakdown
=========================================
Generated: 3 April 2026


1. WHAT IT IS
-------------
This is a personalised CV and job-application platform for Ben Howard. It is a
hybrid system that combines:

  - A public-facing website hosted on GitHub Pages at checkloops.co.uk
  - A local admin toolchain that runs on Ben's Mac
  - AI-powered content generation using OpenAI
  - Multi-source job search aggregation (Indeed, Reed, NHS Jobs, Adzuna)
  - Supabase as a cloud database and file storage backend

The core workflow is: find a job → paste the advert → AI generates a tailored
CV page → review → publish to both GitHub Pages and Supabase → share a unique
URL with each employer.

How cv.html works:
  It is a single HTML page with embedded CSS. When loaded with ?ref=<slug>,
  cv-runtime.js fetches the matching JSON from /data/<ref>.json (or from the
  local API when running locally), then populates every section. It uses gen*
  prefixed fields (AI-generated personalised content) with fallback builder
  functions for each section. Fields like genHeroPositioning,
  genPersonalisedOpening, genWhyThisCompany etc. are the AI outputs; the
  runtime has handwritten fallback generators for when AI content is missing.

Design of cv.html/cv-qr.html:
  Premium editorial design using Cormorant Garamond + Manrope fonts,
  gold/cream colour palette, with sections for: hero positioning, personalised
  opening, why this company, why this role, focus areas, experience mappings,
  evidence examples, 90-day plan, and closing summary.

2c. Local Python Server
~~~~~~~~~~~~~~~~~~~~~~~
local_server.py — a ThreadingHTTPServer that:
  - Serves all static files from the repo root
  - Provides a full REST API under /api/
  - Blocks access to sensitive files (secrets.local.json, local-cache/)
  - Redirects root / to /local-admin/jobspy.html
  - Binds to 127.0.0.1 locally, 0.0.0.0 in Docker


3. WHY IT USES A LOCAL SERVER
------------------------------
The local server exists because several operations cannot run in a browser:

  1. Secret management
     API keys for OpenAI, GitHub, Supabase, Reed, Adzuna, and Indeed are stored
     in macOS Keychain or secrets.local.json. The server reads these securely
     and never exposes them to the frontend.

  2. OpenAI API calls
     The advert parsing and personalised content generation require server-side
     calls to OpenAI's API with the secret key.

  3. GitHub API integration
     Publishing applications requires authenticated GitHub Contents API calls
     (PUT files, update index, create redirect pages). This uses a personal
     access token.

  4. Supabase service-role operations
     Writing to Supabase requires the service-role key (not the anon key),
     which must never be exposed client-side.

  5. Multi-source job search
     Fetching from Indeed (GraphQL API), Reed (REST API), Adzuna (REST API),
     and NHS Jobs (via Supabase edge function) requires server-side API calls
     with credentials.

  6. PDF generation
     Uses headless Chrome (--headless=new --print-to-pdf) to convert CV HTML
     to PDF.

  7. Evidence bank processing
     The CSV evidence bank is loaded and scored server-side using token-matching
     algorithms.

  8. CV profile extraction
     Parses BH CV.html to extract role titles, skills, and sectors for job
     matching/scoring.


4. HOW SUPABASE IS USED
-------------------------
Project URL: https://jntpyqguonknixyksqbp.supabase.co

4a. Database Tables (from supabase/schema.sql)

  applications
    Stores full application data as JSONB with indexed columns for ref,
    company, role, location.
    RLS: Public read; write via service-role key only.
    Primary key: ref (the slug).
    The "application" JSONB column stores the entire application payload.
    Has an auto-updating updated_at trigger.
    The short_code column stores the 8-character shortcode for short URLs.

  reviewed_jobs
    AI-reviewed job adverts with strengths, weaknesses, dealbreakers.
    RLS: Public read; write via service-role key only.
    Primary key: id (UUID).
    Unique constraint on fingerprint.

  cv_contact_requests
    Contact form submissions from cv.html (direction, sender info, message).
    RLS: Anon insert allowed; no public read.

4b. Storage

  Bucket: cv-files (public)
  Path pattern: downloads/<filename>.html
  Used to host downloadable CV files (HTML format) that can be linked from
  the CV page.

4c. Edge Functions

  search-nhs-jobs — called by the local server to proxy NHS Jobs search queries.

4d. Auth/Keys

  Anon key: Hardcoded in local-admin/admin.js for client-side reads and NHS
    search edge function calls.
  Service-role key: Stored in secrets.local.json (gitignored) for all write
    operations.
  All write operations happen server-side through the local Python server.

4e. Data Flow with Supabase

  When an application is published:
    1. Local server upserts the full application JSON to the applications table.
    2. The application JSONB column contains everything needed to render the
       CV page.
    3. The public cv.html can then load application data from either the static
       data/<ref>.json file OR from Supabase (if accessed through the local
       server).


5. THE APPLICATION PIPELINE (END-TO-END)
-----------------------------------------

Step 1: Find a Job
  Use jobspy.html to search across Indeed, Reed, NHS Jobs, and Adzuna
  simultaneously. Jobs are:
    - Normalised into a common format via normalise_search_job()
    - Deduplicated via merge_duplicate_jobs() using title+company fingerprint
    - Scored against Ben's CV profile via score_job() (0-100 scale with tiers:
      Top Match 80+, Strong Match 60+, Worth Reviewing 40+, Low Match)
    - Scored for location proximity, salary, keywords, employer type
    - Optionally AI-reviewed via the "Review" button (calls OpenAI to compare
      against CV)

Step 2: Paste the Advert
  In local-admin/index.html, paste the raw job advert text.

Step 3: AI Generation (/api/generate)
  The server runs a two-stage pipeline via content_generation.py:

  Stage 1 — Advert Extraction:
    Sends the raw text to OpenAI with the ADVERT_EXTRACTION_SYSTEM_PROMPT.
    Returns structured JSON with fields like companyName, roleTitle, sector,
    essentialRequirements, keyFocusAreas, etc.
    Model: configurable, defaults to gpt-5.4-mini.

  Stage 2 — Personalised Content Generation:
    1. Loads the evidence bank from ben_evidence_bank_template.csv
    2. Tokenises the application fields and scores each evidence row using
       Jaccard-style overlap
    3. Selects top 5 evidence examples with diversity bonuses (varies employer,
       sector, proof tags)
    4. Sends the parsed application + selected evidence to OpenAI with the
       SYSTEM_PROMPT
    5. Returns personalised content: hero positioning, opening statement, why
       this company, why this role, experience mappings, focus areas, 90-day
       plan, closing summary, etc.

Step 4: Review
  The admin UI displays all extracted and generated fields in a review grid.
  The user can inspect everything before publishing.

Step 5: Publish (/api/publish)
  When confirmed, the server:
    1. Generates an 8-character short code if needed
    2. Saves to local-cache/data/<ref>.json (always)
    3. Upserts to Supabase applications table (if configured)
    4. Pushes to GitHub via Contents API:
       - data/<ref>.json — the full application JSON
       - data/applications.json — updated index
       - r/<shortcode>/index.html — redirect page
    5. Returns the live URLs (full page, QR page, short URL)

Step 6: Share
  The employer receives a URL like
  https://checkloops.co.uk/cv.html?ref=<slug> or a short URL
  https://checkloops.co.uk/r/<shortcode>/. Both resolve to a beautifully
  designed, personalised CV page.


6. THE EVIDENCE BANK
----------------------
ben_evidence_bank_template.csv contains 15 real career achievement examples
structured as:
  - Situation / Context → Problem / Challenge → What I Did → Result / Outcome
  - Each tagged with: Proof Tags, Sector Tags, Role Tags
  - Each has a Short Version, Headline Version, and metric/number
  - Covers: NHS primary care, manufacturing (Copper Alloys), theme parks
    (Merlin/Alton Towers), Experian, self-employed work, CheckLoops

The matching algorithm in select_evidence_examples():
  1. Builds token sets from application fields (sector, roleTitle,
     keyFocusAreas, etc.)
  2. Scores each evidence row on token overlap (Jaccard-style)
  3. Adds bonuses for public-safe entries and entries with clear metrics
  4. Uses greedy selection with diversity bonuses to avoid repeating the same
     employer/sector/proof type


7. JOB SEARCH SCORING SYSTEM
------------------------------
The score_job() function in local_server.py uses:

  Priority rules (from job-search-preferences.json):
    GP Practice Leadership:            weight 90
    Operations / Transformation:       weight 60
    Digital / Systems Improvement:     weight 42

  CV keyword matching:    4 points per keyword match from the CV profile
  Role title overlap:     6 points when 2+ significant words from CV role
                          titles appear
  Salary scoring:         +4 to +12 points above floor, -45 below floor
  Location fit:           Priority terms scored by position in list,
                          remote/hybrid bonuses
  Exclusion rules:        -80 for clinical roles (nurse, physiotherapist, etc.)
                          unless management is also present
  Employer type bonuses:  GP Practice/Primary Care +20, NHS Trust +12


8. SECRET MANAGEMENT
----------------------
Secrets are loaded via load_local_config() with a three-tier fallback:

  1. macOS Keychain — via "security find-generic-password" (services like
     benhowardcv-github-token, benhowardcv-openai-api-key, etc.)
  2. secrets.local.json — local config file (gitignored)
  3. Environment variables — GITHUB_TOKEN, OPENAI_API_KEY, SUPABASE_URL, etc.

11. QR CODE SYSTEM
-------------------
Uses vendor/qrious.min.js (a QR code generation library). When an application
is published, the admin UI generates a QR code pointing to the short URL. This
QR code can be:
  - Printed on a physical CV (via BH CV.html)
  - Displayed in the admin result panel
  - Used in the downloadable CV files


12. THE BASE CV
----------------
BH CV.html is a traditional two-page print-format CV (A4 paper size,
210mm x 297mm) with:
  - Two-column layout (main content + sidebar)
  - Employment history, skills, sector experience
  - Used as the source for extract_cv_profile() which parses out role titles,
    skills, and sectors for job matching
  - QR codes on printed versions link back to the tailored online CV page

15. LAUNCHER
--------------
"OPEN THIS - Ben Howard CV.command" is a macOS shell script that:
  1. Finds a free port starting from 8000
  2. Starts the local Python server in the background
  3. Waits for the server to be ready (polls /api/status)
  4. Opens the admin URL in the default browser
  5. Cleans up the server process on exit

This is a comprehensive, self-contained job application system combining static
hosting, AI generation, multi-source job search, and cloud persistence into a
workflow tailored to one person's job search.
